Scoring du risque d'escalade de conflit géopolitique

Du nettoyage des données à une API de scoring déployable : cycle complet MLOps sur un panel mensuel de 20 pays, avec score métier, comparaison de modèles, interprétabilité et surveillance du drift.

Dépôt : github.com/gedeon92/ExamensML Dataset : 1 320 lignes · 20 pays · 2020-01 → 2025-06 Cible : conflict_escalation_6m (binaire)
Objectifs

Prédire, chiffrer, déployer

Construire un modèle de scoring binaire qui anticipe une escalade de conflit dans les 6 mois, en pondérant explicitement le coût d'une alerte manquée contre celui d'une fausse alerte — puis l'industrialiser : API, interface, suivi MLflow, et surveillance de la dérive dans le temps.

Méthode

Le pipeline, étape par étape

  1. 01

    Environnement MLflow

    Tracking local (SQLite), expérience dédiée, run de test pour valider la journalisation avant l'entraînement des modèles.

  2. 02

    Préparation des données

    Nettoyage, correction d'un bug de valeur sentinelle (last_conflict_year = 0), détection et retrait de deux variables de fuite (instability_score, political_stability_index), split chronologique train/test.

  3. 03

    Score métier

    Une escalade non détectée coûte 5× plus qu'une fausse alerte — logique d'alerte précoce, transposée d'un scoring de risque de crédit.

  4. 04

    Modélisation

    Logistic Regression, Random Forest, XGBoost — GridSearchCV optimisé sur le score métier, SMOTE appliqué dans les folds d'entraînement uniquement, interprétabilité SHAP + LIME.

  5. 05

    API de scoring

    FastAPI, tests automatisés, image Docker, pipeline CI/CD GitHub Actions.

  6. 06

    Interface Streamlit

    Saisie manuelle des indicateurs d'un pays, appel à l'API, résultat visuel avec niveau de risque.

  7. 07

    Data drift

    PSI, test de Kolmogorov-Smirnov, Chi², dérive des prédictions — et une stratégie de réentraînement.

Résultats

Random Forest l'emporte sur le score métier

Les trois modèles battent largement la stratégie naïve « ne jamais alerter ». Random Forest obtient le meilleur score métier ; XGBoost a le meilleur ROC AUC mais un seuil par défaut moins bien calibré pour notre coût 5:1.

Déploiement

De l'expérimentation au service

API FastAPI Testée

Endpoints /health et /predict, feature engineering partagé avec le notebook, validation Pydantic stricte. Tests pytest + CI GitHub Actions verts.

Interface Streamlit Testée

Formulaire de saisie manuelle par pays/mois, appel direct à l'API, probabilité affichée avec message de risque coloré.

CI/CD Vert

Tests automatiques à chaque push/PR ; déploiement continu déclenché par webhook une fois un service cloud connecté.

Déploiement cloud À finaliser

Image Docker prête (Render, Fly.io, AWS, GCP...) — nécessite la création d'un compte/service cloud côté équipe projet.

Surveillance

Data drift & réentraînement

Sur la période de test (2024-06 → 2025-06) simulant un batch récent : dérive globalement limitée. protest_acceleration et arms_imports_index montrent une dérive modérée à surveiller ; la dérive des scores prédits (PSI = 0,19) reste juste sous le seuil d'alerte.

Indicateurs suivis

PSI & test KS par feature, Chi² sur les catégorielles, PSI des scores prédits (disponible en temps réel, contrairement aux étiquettes retardées de 6 mois).

Stratégie de réentraînement

Réentraînement trimestriel programmé + déclenchement anticipé si PSI > 0,2 ou chute du score métier réel sous le baseline de déploiement (0,582) — promotion en champion/challenger via MLflow.

Perspectives

Ce qui viendrait ensuite

Déploiement cloud réel

Connecter le webhook CI/CD à un service (Render/Fly.io/AWS/GCP) pour une API réellement accessible en ligne.

Registre de modèles MLflow

Formaliser la promotion champion/challenger avec le Model Registry plutôt qu'un simple tag.

Boucle de monitoring automatisée

Job planifié qui recalcule PSI/KS à chaque nouveau batch et alerte l'équipe (Slack/email) sans intervention manuelle.

Revue humaine des alertes critiques

Pour les prédictions à risque « élevé », une validation par un analyste avant toute décision opérationnelle.
